Why do we need redux?

Redux:

Redux manages the state of an app using a single store. To modify the state, an action is dispatched to the store and a reducer updates the store based on the type and other values of the action. In order to use the new state, the store can be subscribed to. Subscribing requires a callback function which is called whenever the store updates.

From <https://github.com/pshrmn/notes/blob/master/redux/redux.md>

  • The redux store(Central, global state that all our component can access) - j-son object - current state of our app
  • Redux actions are j-son objects, consisting of two things, an action type, which is basically just a string naming the action, and a payload of additional data. The purpose of redux actions, is to explicitly define the different events that can occur in our application. So for example, we could have an action with the type: user data loaded, with a payload that contains the actual user data that we just fetched from the server.
  • A reducer updates the state of the store. A reducer takes two arguments, the current state and the action that has been dispatched. The current state is manipulated and returns a new state based on the action.

^ The action is reduced to get the new updated state and the components in our app get read only access to that updated state

In programming and software development, the payload is used in the context of message protocol to differentiate between the assisting and actual data in a query string. For instance, Let’s consider this JSON web service response.

{

"status":"OK",

"data":

{

"message": "Welcome, world!"

}

}

In the above example, the payload is the Welcome, World! Since it is the part of the query string that the user is interested in. The rest of the information is referred to as the overhead data. This is because it is only used to show the source or destination and display authenticity.

After creating the actions and reducer you have to connect everything.

Here are two interesting functions:

  • mapStateToProps: The state argument that gets passed to mapStateToProps is an object that represents the entire Redux state.
  • Now the job of mapStateToProps is to take this state object and return another object containing the pieces of that state that our component needs access to.
  • mapDispatchtoProps. This function works in a similar way to mapStateToProps, and in the properties of the object to be returned will be passed to our component as props. The difference here is that instead of taking the Redux state as an argument, it takes something called dispatch. Now what dispatch is is it's a function that allows our components to trigger actions that our Redux store will respond to.
  • The only way to update the state is to call store.dispatch() and pass in an action object.
  • You can think of dispatching actions as "triggering an event" in the application. Something happened, and we want the store to know about it. Reducers act like event listeners, and when they hear an action they are interested in, they update the state in response.

Redux Thunk:

In redux, a normal dispatch take in an object with a type and a payload but redux thunk takes in an asynch function.

A thunk is simply a function that returns another function, which contains the actual logic that we want to perform when it's triggered. Thunks can be dispatched the same way as actions.

Selectors(functions)

  • Selectors give us a place to put logic for combining, filtering, transforming storing data
  • selectors are meant to abstract away how data is stored in redux as well as give us a place to put logic for transforming redux data into data that our components can use.
  • higher-order selectors that we've defined are incomplete and completed to-do's, have no idea how our data is formatted in the redux store. Nowhere here are we referring to state.todos.data or state.todos.isLoading. So, even our higher-order selectors rely on our lower-level selectors, such as getTodos, to worry about the exact format of our data in redux
  • the selectors we create with Reselect's createSelector function use something called memoization(save computational power so it doesn't rerun the function if the arg we pass in is the same as before)
  • resultFunc() -> reference to the last func that we passed to createSelector

Class component vs function component

A functional component is just a plain JavaScript function which accepts props as an argument and returns a React element.

A class component requires you to extend from React.Component and create a render function which returns a React element.

  • Class can use setState, this, and need render()
